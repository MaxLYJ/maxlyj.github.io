## ‚úÖ Fixed in iteration 9 (SEO: JSON-LD structured data on JS-rendered project pages)

### SEO
- **Added schema.org `Article` JSON-LD structured data to all four project pages (part of P1 #1).** `project-instance-loader.js` builds each project page's entire `<body>` at runtime (fetch `template-content.html` + per-project JSON, construct DOM in JS), so search engines that fetch-and-parse without executing JavaScript see an empty document ‚Äî only the `<head>` meta tags existed. Google *does* eventually render JS, but that is a deferred, secondary pass; structured data in the static HTML is parsed on the first, immediate crawl. Each page (`farcry-6.html`, `division-2.html`, `d-walker-vs-sahelanthropus.html`, `raiden-vs-gekko.html`) now carries a `<script type="application/ld+json">` block in its `<head>` with the project's real facts: `headline` (canonical project title), `description` (the same polished copy as the page's `<meta name="description">`), `image` (the cover, identical to `og:image`), `url` (identical to `<link rel="canonical">`), `mainEntityOfPage` (`@id` = canonical), `author` and `publisher` (`Person` ‚Üí Yujia Max Liu, self-published portfolio), and `keywords` (the real tool/language/role terms from the config ‚Äî e.g. *Houdini, Substance Designer, VEX, HLSL* for Far Cry 6). ## ‚úÖ Fixed in iteration 12 (SEO: static pre-render fallback for JS-rendered project pages ‚Äî P1 #1, option a)

The headline SEO gap from iteration 1: every project page builds its entire `<body>` at runtime in `project-instance-loader.js`, so a no-JS fetch (and a crawler's immediate first pass, before Google's deferred JS-rendering pass) saw an empty document ‚Äî only the `<head>` meta + the iteration-9 JSON-LD carried the project's facts. ### Architecture
- **Added `scripts/prerender-project-pages.js`** ‚Äî a dependency-free Node script (run pre-deploy, `node scripts/prerender-project-pages.js` from the repo root) that reads each project config (`Resources/Project Instances/config/<slug>.json`) + `data/taxonomy.json` and inlines the project's **real content** as semantic HTML inside a `<noscript>` block in each of the four project pages. The generated block is delimited by `BEGIN/END project-noscript` marker comments; the script strips any prior generation **and** the legacy bare `This page requires JavaScript` `<noscript>` before re-inserting ahead of `<script src="shared.js">`, so it is **idempotent** (re-running produces byte-identical output ‚Äî verified). It reads `CDN_BASE` from `shared.js` (regex) so the build tool and the runtime share one source of truth, and fails loudly if the constant can't be found.
- **What gets inlined per page:** a back-to-home link (no site nav exists without JS ‚Äî the loader injects the sidebar/top bar too), kicker, `<h1>` title, description, the cover image, a `<dl>` metadata grid (Tools / Languages / Period / Role / Tags resolved from the taxonomy), **every** `projectDetails.blocks` entry rendered to its semantic equivalent (`h2/h3/p` as-is; `image` ‚Üí `<figure><img>` with alt; `image-compare` ‚Üí a `<figure>` of both before/after images + caption; `video` ‚Üí a text link to the source URL), and a closing note explaining the interactive version needs JS. This is substantial, real, crawlable body text (e.g. division-2 inlines 40 blocks / 23 figures).
- **Added a `.project-noscript` style block to `style.css`** (‚âà155 lines, design-token-based: `--text/--muted/--accent/--accent-2/--panel/--panel-2/--line`, responsive at ‚â§640px) so the text fallback reads like the rest of the site for no-JS visitors. It only ships in `style.css` (always loaded); it is inert for JS visitors because `<noscript>` content is never rendered when scripting is on.

### Design decision ‚Äî `<noscript>`, not primary visible `<body>`
The review's option (a) literal text says "inline into the page `<body>`." A primary-body pre-render (content inside `[data-project-instance-root]`, which the loader clears on success) would carry marginally more SEO weight, but it has two failure modes I **cannot verify without a browser** (there is no build/test system per `AGENTS.md`, no jsdom/headless browser): (1) a flash of unstyled pre-rendered content for JS visitors during the async fetch window (content appears, is wiped, then re-rendered ‚Äî worse than the current blank‚Üírender), and (2) any edge case where the content isn't cleared risks duplicating it in the rendered DOM. The `<noscript>` approach has **none** of these risks: the HTML5 parser treats `<noscript>` content as RAWTEXT when scripting is enabled (it is never added to the DOM as elements, so it cannot duplicate, flash, or interfere with the loader's `querySelector`/`innerHTML`), and renders it as normal flow content for no-JS visitors and non-JS crawlers. Its only failure mode is "does nothing extra" ‚Äî the principled call when you can't smoke-test. Google indexes `<noscript>` content; combined with the existing per-page meta + JSON-LD, the pages now have comprehensive static SEO coverage. A primary-body SSR/hydration refactor remains the longer-term enhancement (noted in P1 #1).

## ‚úÖ Fixed in iteration 17 (performance: cut wasteful featured-carousel image preloading ‚Äî P3 #13)

The featured carousel's preloading strategy warmed far more than the LCP needed, competing with first paint on slow connections. Two distinct defects, both in `home.js`, fixed together as one cohesive "stop fetching images you don't need yet" change. Chosen over the iteration-16 candidates per the objective's priority order (performance before content/architecture): P4 #17 (contact CTA) is **blocked** ‚Äî no email exists anywhere in the repo and fabricating one is out of scope ‚Äî and P4 #16 (skill-tag/taxonomy cross-check) is mostly documentation since the two lists are genuinely *different dimensions* (taxonomy = competency areas for filtering the works gallery; About `.skill-tag`s = concrete tools/engines); P3 #13 is a concrete, verifiable performance code change.

### Performance ‚Äî neighbor warming reduced to cover-only (P3 #13)
- **`warmNeighborPages()` ‚Üí `preloadPageAssets()` previously fetched the cover + every thumbnail + every HighRes "full" for *both* adjacent carousel slides on every `renderPage()`** ‚Äî up to ~18 speculative image requests (1 cover + up to 4 thumbs + up to 4 HighRes fulls, √ó 2 neighbors), several of them large HighRes assets shown only on hover. When a neighbor becomes the current slide its thumbnails (small LowRes) render and fetch on demand, and the HighRes full swaps in on hover exactly as it already did for the current slide ‚Äî the current slide's fulls were never preloaded (only neighbor fulls were), so this makes neighbors *consistent* with the current slide rather than regressing anything.

### Code quality ‚Äî removed two redundant same-URL preloads (new finding)
- **`renderPage()` called `preloadImage(page.cover)` immediately after `frMainImage.src = page.cover`, and `preloadImage(thumbPath)` immediately after `thumbImage.src = thumbPath` inside the thumbnail render loop.** Both duplicated a fetch the visible DOM `<img>` was already making ‚Äî setting `.src` on a real `<img>` starts the request, so a `new Image()` with the same URL is a redundant second request the browser then has to dedup. Removed both, leaving a brief comment at each site explaining *why* no explicit preload is needed (so a future contributor doesn't "restore" them for performance ‚Äî same anti-re-introduction rationale as the iteration-10 dead-code deletions). `preloadImage` is now called exactly once (the neighbor-cover warm), so no helper was orphaned.

### Net effect
On initial homepage load the carousel now speculatively fetches **2 neighbor covers** instead of up to ~18 images (cover + thumbs + fulls √ó 2 neighbors) plus up to 5 redundant same-URL duplicates (1 cover + up to 4 thumbs on the current slide). Carousel navigation still feels instant (neighbor covers are warm); thumbnails/fulls fetch on demand when actually viewed. **Zero functional behavior changed** ‚Äî only *which* images are requested ahead of time.
