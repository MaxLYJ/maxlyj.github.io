## ✅ Fixed in iteration 5 (accessibility: image-compare slider is keyboard-operable)

### Accessibility (a11y)
- **The image-compare slider is now operable and announced without a mouse (P2 #7).** `initImageCompareSliders` was pointer-only (mousedown/touchmove). ## ✅ Fixed in iteration 6 (architecture: extract `shared.js` — P5 #19, part 1)

### Architecture / maintainability
- **Extracted the byte-identical helpers duplicated across `home.js` and `project-instance-loader.js` into a new `shared.js`** (P5 #19). `CDN_BASE`, `toCdnUrl()`, `loadVersionTag()`, and `TAXONOMY_MANIFEST_PATH` were defined verbatim in both files. Classic scripts share the global lexical scope, so top-level `const`/`function` in `shared.js` are visible to the subsequently-loaded entry script without `import`/`export` or a module-system migration; the duplicate declarations were removed from both consumers, so there is no "Identifier has already been declared" collision. Moved the call to *after* `root.append(header, overlay, sidebar, main)`, where the injected top bar is guaranteed to be in the DOM, so project pages now show their commit version too.

### Scope note — what was *not* extracted this iteration

Each page (`farcry-6.html`, `division-2.html`, `d-walker-vs-sahelanthropus.html`, `raiden-vs-gekko.html`) now carries a `<script type="application/ld+json">` block in its `<head>` with the project's real facts: `headline` (canonical project title), `description` (the same polished copy as the page's `<meta name="description">`), `image` (the cover, identical to `og:image`), `url` (identical to `<link rel="canonical">`), `mainEntityOfPage` (`@id` = canonical), `author` and `publisher` (`Person` → Yujia Max Liu, self-published portfolio), and `keywords` (the real tool/language/role terms from the config — e.g. *Houdini, Substance Designer, VEX, HLSL* for Far Cry 6). ## ✅ Fixed in iteration 12 (SEO: static pre-render fallback for JS-rendered project pages — P1 #1, option a)

The headline SEO gap from iteration 1: every project page builds its entire `<body>` at runtime in `project-instance-loader.js`, so a no-JS fetch (and a crawler's immediate first pass, before Google's deferred JS-rendering pass) saw an empty document — only the `<head>` meta + the iteration-9 JSON-LD carried the project's facts. Closed with a build-time pre-render.

### Architecture
- **Added `scripts/prerender-project-pages.js`** — a dependency-free Node script (run pre-deploy, `node scripts/prerender-project-pages.js` from the repo root) that reads each project config (`Resources/Project Instances/config/<slug>.json`) + `data/taxonomy.json` and inlines the project's **real content** as semantic HTML inside a `<noscript>` block in each of the four project pages. The generated block is delimited by `BEGIN/END project-noscript` marker comments; the script strips any prior generation **and** the legacy bare `This page requires JavaScript` `<noscript>` before re-inserting ahead of `<script src="shared.js">`, so it is **idempotent** (re-running produces byte-identical output — verified). It reads `CDN_BASE` from `shared.js` (regex) so the build tool and the runtime share one source of truth, and fails loudly if the constant can't be found.
- **What gets inlined per page:** a back-to-home link (no site nav exists without JS — the loader injects the sidebar/top bar too), kicker, `<h1>` title, description, the cover image, a `<dl>` metadata grid (Tools / Languages / Period / Role / Tags resolved from the taxonomy), **every** `projectDetails.blocks` entry rendered to its semantic equivalent (`h2/h3/p` as-is; `image` → `<figure><img>` with alt; `image-compare` → a `<figure>` of both before/after images + caption; `video` → a text link to the source URL), and a closing note explaining the interactive version needs JS. This is substantial, real, crawlable body text (e.g. division-2 inlines 40 blocks / 23 figures).
- **Added a `.project-noscript` style block to `style.css`** (≈155 lines, design-token-based: `--text/--muted/--accent/--accent-2/--panel/--panel-2/--line`, responsive at ≤640px) so the text fallback reads like the rest of the site for no-JS visitors. It only ships in `style.css` (always loaded); it is inert for JS visitors because `<noscript>` content is never rendered when scripting is on.

### Design decision — `<noscript>`, not primary visible `<body>`
The review's option (a) literal text says "inline into the page `<body>`." A primary-body pre-render (content inside `[data-project-instance-root]`, which the loader clears on success) would carry marginally more SEO weight, but it has two failure modes I **cannot verify without a browser** (there is no build/test system per `AGENTS.md`, no jsdom/headless browser): (1) a flash of unstyled pre-rendered content for JS visitors during the async fetch window (content appears, is wiped, then re-rendered — worse than the current blank→render), and (2) any edge case where the content isn't cleared risks duplicating it in the rendered DOM. The `<noscript>` approach has **none** of these risks: the HTML5 parser treats `<noscript>` content as RAWTEXT when scripting is enabled (it is never added to the DOM as elements, so it cannot duplicate, flash, or interfere with the loader's `querySelector`/`innerHTML`), and renders it as normal flow content for no-JS visitors and non-JS crawlers. Its only failure mode is "does nothing extra" — the principled call when you can't smoke-test. Google indexes `<noscript>` content; combined with the existing per-page meta + JSON-LD, the pages now have comprehensive static SEO coverage. A primary-body SSR/hydration refactor remains the longer-term enhancement (noted in P1 #1).

## ✅ Fixed in iteration 22 (mobile polish + dark-mode signaling: `theme-color` + `color-scheme`)

The explicit backlog was depleted to owner-driven / browser-gated items, so this was another fresh-audit pass — over **browser-chrome theming and dark-mode signaling**, the last unimplemented standard `<head>` surface (iteration 21 noted `theme-color` as a clean candidate and flagged `color-scheme` as a productive fresh-audit angle in its learnings). A repo-wide grep confirmed the site shipped **no `<meta name="theme-color">` and no `<meta name="color-scheme">` on any page** (and `style.css` declared neither), despite being an unconditionally dark site with no `prefers-color-scheme` media query. Both gaps are static HTML, fully verifiable without a browser, on-brand, and zero-risk — so they were fixed together as one cohesive "tell the browser this is a dark site" unit.

### Why both matter (they solve different problems)
- **`<meta name="theme-color" content="#0f111a">`** colors the mobile browser chrome — the Android-Chrome address bar and the Safari-mobile browser-chrome / status-bar area — to match the page background. On a dark portfolio the default white browser chrome above a dark page is a visible polish gap; `theme-color` makes the chrome blend into the site. The value is the page background token `--bg` (`#0f111a`), identical on all pages.

Declaring `color-scheme: dark` makes the browser paint a dark canvas from the very first frame and theme scrollbars dark. This is a genuine UX/correctness improvement, not just mobile chrome.
- They are complementary, not redundant: `theme-color` is purely cosmetic chrome coloring; `color-scheme` governs UA rendering behavior. MDN best practice is to declare `color-scheme` in **both** the `<meta>` (parsed before CSS, governs the pre-CSS-load canvas + UA UI) and the CSS `:root` (the canonical declaration, applies once CSS is parsed) — both were added.

## ✅ Fixed in iteration 23 (accessibility: keyboard navigation broken on carousel thumbnails)

The explicit backlog was depleted to owner-gated / browser-gated items, so — per the iteration-22 learning — this was a **deeper runtime/correctness review** of the carousel interaction code. It surfaced one concrete, spec-confirmed, high-impact accessibility defect: **the featured-carousel thumbnails could not be activated by keyboard.** This is exactly the kind of bug a senior reviewer flags because it is invisible in a mouse-only click-through yet strands any keyboard / screen-reader user.

### The bug — `preventDefault()` on Enter/Space cancelled the native click
Each carousel thumbnail is a native `<button class="fr-thumb">` (`home.js`, `renderPage()`). It carried *two* activation paths:
- a `click` handler that swaps the hero preview AND navigates: `window.location.href = page.targetUrl;` (the carousel's primary action — go to the project), and
- a `keydown` handler that, on `Enter` or `" "`, called `event.preventDefault()` then `activateThumb()`.

The `keydown` handler was the bug. A native `<button>` fires its synthetic `click` event **as the default action of the keydown/keyup** for Enter/Space (Enter fires the click on `keydown`; Space fires it on `keyup`). `event.preventDefault()` on that keydown **cancels the default action, so the native `click` never fires.** Net effect: a keyboard user who focused a thumbnail and pressed Enter or Space got only the hero-preview swap (which the `focus` handler already performed) and **never navigated** — the thumbnail's primary action was unreachable by keyboard. A mouse click worked (click is a separate event unaffected by the keydown `preventDefault`), so the defect was invisible unless you navigated by keyboard. (Confirmed against the HTML activation model: *"For a native `<button>`, Enter triggers activation on keydown, Space on keyup; calling `preventDefault()` during the keydown cancels the native activation and thus the synthetic click."*)

### Internal proof it was a defect, not an intentional pattern
The **loader's own gallery thumbnails** (`.project-thumb`, `project-instance-loader.js` ~line 405–407) use a **click-only** pattern — no manual Enter/Space `keydown` handler at all — and keyboard users activate them fine (native Enter/Space → click → `renderProjectGallery`). That is the correct idiom for a native button. The carousel was the *only* place in the codebase that added the extra `keydown` + `preventDefault`, which can only *suppress* the native click the loader relies on. The divergence itself is the proof: click-only is sufficient for keyboard (the loader proves it); the carousel's extra handler could only add harm.

### Fix — removed the redundant + harmful keydown handler
Deleted the thumbnail `keydown` block entirely. Enter/Space now fall through to the native button activation → `click` → `activateThumb()` + navigate, restoring keyboard parity with the mouse. Nothing else changed: the `focus`/`blur` preview-swap, `mouseenter`/`mouseleave`, the carousel-wide ArrowLeft/ArrowRight slide navigation, and the mouse/touch click paths are all untouched. Added a comment at the site explaining *why* no Enter/Space handler exists (so a future contributor doesn't "restore" the broken pattern — same anti-re-introduction rationale as the iteration-10/17 deletions).

### Why this runtime change is safe without a browser smoke-test
Unlike the iteration-19 `a > button` nested-interactive finding (a *structural* DOM change that genuinely wants a browser check), this is a **logic fix that restores native semantics** — the same class of provably-safe change as iteration 17's preload-trimming:
- A native `<button type="button">` is *specified* to fire `click` on Enter/Space. Removing a JS handler that suppressed that cannot break activation; it can only restore it.
- The `click` handler is unchanged and covers every activation path (mouse, touch, Enter, Space).
- The only behavior delta is that keyboard Enter/Space now navigates (matching the mouse) instead of doing nothing — strictly an improvement, with no regression for any input method (verified: Space on a focused button does not scroll, so no page-scroll side effect returns).

## ✅ Fixed in iteration 24 (runtime review: version-tag external-API path + completing the `preventDefault` audit)

Continuing the **deeper runtime/correctness review** pivot that paid off in iteration 23 (the iteration-22 learning said the static-`<head>`/site-completeness surfaces are exhausted and the productive remaining angle is spec-grounded JS logic bugs invisible to a mouse click-through). Iteration 23's learning explicitly named two iter-24 candidates — *"the remaining `preventDefault`/event-ordering sites"* and *"the `loadVersionTag` GitHub-API failure path"* — and both were taken this iteration. Both are provably-safe logic fixes (no structural/DOM change), fully verifiable without a browser.

### Robustness / performance — `loadVersionTag` now caches the commit SHA per session
- **The defect:** `loadVersionTag()` (`shared.js`) populates the decorative `.top-bar-version` pill by fetching `https://api.github.com/repos/MaxLYJ/maxlyj.github.io/commits?…` — the GitHub REST API, **unauthenticated**, called once on the homepage (`home.js:11`, module load) **and again on every project page** (`project-instance-loader.js:632`, after the template injects the top bar). A visitor browsing the homepage + the 4 project pages therefore made **5 unauthenticated GitHub API calls per session**. GitHub's unauthenticated rate limit is **60 requests/hour per originating IP** (shared across all visitors behind the same NAT/corporate proxy/CGNAT), so this amplified a fragile external dependency 5× — and on every repeat call the pill sat empty until the (rate-limited or slow) round-trip resolved, with no cross-page memory that a *prior* page in the same session had already fetched the SHA.
- **The fix:** cache the resolved label in `sessionStorage` and reuse it across pages within the same browsing session. `loadVersionTag()` now reads `sessionStorage.getItem("maxlyj:version-tag")` first; on a hit it paints the pill synchronously and returns (no network). On a miss it fetches as before, and on success writes `sessionStorage.setItem(...)` so the next page in the session hits the cache. This collapses the per-session call count from **5 → 1**: the first page fetches, the other four read the cache and show the version instantly (and never leave the pill empty on a rate-limited repeat).
- **Why `sessionStorage`, not `localStorage`:** sessionStorage is scoped to the tab and cleared when the tab closes, so visitors reaching the site in a *new* session after a deploy always fetch the fresh SHA rather than reading a stale one persisted across days. The staleness window is at most one browsing session — exactly right for a decorative "current version" pill.
- **Conservative fallback (same pattern as `prefersReducedMotion`):** every `sessionStorage` access is wrapped in its own `try/catch` — `getItem` failure → `cached = null` (falls through to fetch); `setItem` failure → leave uncached (next page refetches). So an unavailable sessionStorage (Safari private mode, disabled storage, the `SecurityError` some embed contexts throw) silently degrades to the original always-fetch behavior. No failure path is *worse* than before; the cache is strictly additive.
- **Negative-cache intentionally NOT added:** a transient failure (rate limit, offline, GitHub 5xx) is not cached, so the next page retries rather than pinning a "could not load" state for the whole session. Positive-cache-only is the conservative, correct choice.

### Code quality — removed a redundant `preventDefault` + `stopPropagation` block (preventDefault audit, complete)
- Iteration 23's learning asked to audit *"the remaining `preventDefault`/event-ordering sites."* The full inventory:

## ✅ Fixed in iteration 30 (runtime correctness: featured-carousel showed the prior slide's cover when a thumbnail was focused/active and the slide advanced)

Iteration 26's end-to-end non-event-ordering read paid off on the *loader*; the analogous read of `home.js` (only its event-ordering had been audited, in iterations 23–24) surfaced a real shipped correctness bug in the featured-carousel thumbnail/revert logic — invisible to the mouse-only click-through prior iterations relied on, like the iter-23 keyboard-activation bug.

### The defect — `renderPage()`'s thumbnail teardown clobbered the new slide's cover with the prior slide's
`renderPage()` advances the carousel: it sets the new slide's title/description/cover, then rebuilds the thumbnail row with `frThumbs.innerHTML = ""`. Each thumbnail carries `mouseleave`/`blur` handlers that revert the main image to **that thumb's slide's cover** (closure variable `page`). When a thumbnail is focused (keyboard: Tab to a thumb) or has been activated and the slide then advances — by ArrowLeft/Right, or any re-render — the teardown (`innerHTML = ""`) destroys that thumb and synchronously fires its `blur`/`mouseleave`. That revert ran **after** `renderPage()` had already set the new slide's cover, so the prior slide's cover won. Net shipped effect: the **new slide's title/description/live-region rendered over the *previous* slide's hero image** — a visible desync on the homepage's primary showcase widget.

### Proof it was a defect (not a pattern), and that the fix works
A token-free logic simulation of the teardown reproduces the bug exactly (focus Farcry thumb → ArrowRight to Division → `frMainImage` ends at `FARCRY_cover`, not `DIVISION_cover` — Division title over the Farcry image). The loader's own `.project-thumb` gallery (project pages) is single-page and never torn down on navigation, so this defect class is unique to the carousel's rebuild-on-slide-change — same "the sibling code is the proof-of-correctness" tell as iter 23.

### Fix — render-token guard on the revert handlers
Extracted the duplicate `mouseleave`/`blur` bodies into one `revertToCover()` closure that **bails when the slide has advanced** (`if (currentToken !== renderToken) return`). `renderToken` is incremented at the very *top* of `renderPage()`, before any DOM teardown, so by the time a teardown-fired revert runs it is always stale → bail → the new slide's cover survives. Same-slide leave/blur (`currentToken === renderToken`) still reverts normally, so there is no regression for the intended hover/focus-off behavior. The simulation confirms both paths: cross-slide teardown preserves `DIVISION_cover` (PASS); same-slide blur reverts to `FARCRY_cover` (PASS).

### Defects found — all genuinely owner-gated (R2 source-asset re-export)

2. **Bytes — division2 cover is 4K / 771 KB.** Served for a ≤1280px hero, a ~330px works card, and the carousel hero. ~3–10× the bytes the display needs. Fix = R2: provide a ~1600px `cover.webp` variant (the `thumb_0N`/`full_0N` split already hints at this — extend it to the cover).

- **No higher-res variant is being ignored** — every "undersized" image has no larger sibling on R2 (probed), so there is no repo-side repoint available; the gap is purely source-asset quality.

### Repo-side fix shipped this iteration (the part that IS fixable without R2)
