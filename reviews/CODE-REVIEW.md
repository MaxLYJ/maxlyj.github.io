## ✅ Fixed in iteration 5 (accessibility: image-compare slider is keyboard-operable)

### Accessibility (a11y)
- **The image-compare slider is now operable and announced without a mouse (P2 #7).** `initImageCompareSliders` was pointer-only (mousedown/touchmove). The slider is live content — it ships on `division-2.html` (5 instances) and `farcry-6.html` (1 instance) via the `image-compare` block. - **Removed a redundant `aria-label` on a role-less element (P2 #8).** `<div class="tag-system-meta" aria-label="Tag system update reminder">` in `index.html` had no role, so the `aria-label` was ignored by AT — and the name ("Tag system update reminder") didn't even describe the visible content. ## ✅ Fixed in iteration 6 (architecture: extract `shared.js` — P5 #19, part 1)

### Architecture / maintainability
- **Extracted the byte-identical helpers duplicated across `home.js` and `project-instance-loader.js` into a new `shared.js`** (P5 #19). `CDN_BASE`, `toCdnUrl()`, `loadVersionTag()`, and `TAXONOMY_MANIFEST_PATH` were defined verbatim in both files. They now live once in `shared.js`, which is loaded as a classic `<script>` **before** the page-specific entry script on every page (`index.html` → before `home.js`; the four project pages → before `project-instance-loader.js`). Classic scripts share the global lexical scope, so top-level `const`/`function` in `shared.js` are visible to the subsequently-loaded entry script without `import`/`export` or a module-system migration; the duplicate declarations were removed from both consumers, so there is no "Identifier has already been declared" collision. - **Fixed a latent version-tag bug exposed by the extraction (bonus correctness).** `project-instance-loader.js` called `loadVersionTag()` at the very top of the file, but `.top-bar-version` lives inside the shared template that is fetched and injected *asynchronously* by `loadProjectInstanceTemplate()` — so on project pages the element did not exist when the call ran, the helper no-op'd, and **the version pill was never populated on any project page** (the homepage was unaffected because its pill is in the static HTML). Moved the call to *after* `root.append(header, overlay, sidebar, main)`, where the injected top bar is guaranteed to be in the DOM, so project pages now show their commit version too.

### Scope note — what was *not* extracted this iteration

The render/swipe bodies are near-identical but the entry points differ.

These remain the suggested next extraction iteration (see plan below).

## ✅ Fixed in iteration 9 (SEO: JSON-LD structured data on JS-rendered project pages)

### SEO

Google *does* eventually render JS, but that is a deferred, secondary pass; structured data in the static HTML is parsed on the first, immediate crawl. Each page (`farcry-6.html`, `division-2.html`, `d-walker-vs-sahelanthropus.html`, `raiden-vs-gekko.html`) now carries a `<script type="application/ld+json">` block in its `<head>` with the project's real facts: `headline` (canonical project title), `description` (the same polished copy as the page's `<meta name="description">`), `image` (the cover, identical to `og:image`), `url` (identical to `<link rel="canonical">`), `mainEntityOfPage` (`@id` = canonical), `author` and `publisher` (`Person` → Yujia Max Liu, self-published portfolio), and `keywords` (the real tool/language/role terms from the config — e.g. *Houdini, Substance Designer, VEX, HLSL* for Far Cry 6). This gives crawlers rich, machine-readable content for the JS-rendered pages without touching the working renderer or adding a build step — the pragmatic SEO win short of full static pre-rendering.
- **Schema-type choice — `Article`, no fabricated dates.** These are authored technical-art case studies, so `Article` (well-supported by Google's rich results) is the most semantically useful type; `SoftwareApplication` was rejected because these are not downloadable apps. `datePublished` was **deliberately omitted** rather than fabricated: the configs carry only a work-period string (`time`, e.g. *"Sep 2020 – Dec 2021"*, *"2022-2024"*), not a page-publication date, and a Technical Director review should not invent dates that Google would surface as freshness signals — that is a real data gap worth filling later (see updated P1 #1). ## ✅ Fixed in iteration 12 (SEO: static pre-render fallback for JS-rendered project pages — P1 #1, option a)

The headline SEO gap from iteration 1: every project page builds its entire `<body>` at runtime in `project-instance-loader.js`, so a no-JS fetch (and a crawler's immediate first pass, before Google's deferred JS-rendering pass) saw an empty document — only the `<head>` meta + the iteration-9 JSON-LD carried the project's facts. Closed with a build-time pre-render.

### Architecture
- **Added `scripts/prerender-project-pages.js`** — a dependency-free Node script (run pre-deploy, `node scripts/prerender-project-pages.js` from the repo root) that reads each project config (`Resources/Project Instances/config/<slug>.json`) + `data/taxonomy.json` and inlines the project's **real content** as semantic HTML inside a `<noscript>` block in each of the four project pages. The generated block is delimited by `BEGIN/END project-noscript` marker comments; the script strips any prior generation **and** the legacy bare `This page requires JavaScript` `<noscript>` before re-inserting ahead of `<script src="shared.js">`, so it is **idempotent** (re-running produces byte-identical output — verified). It reads `CDN_BASE` from `shared.js` (regex) so the build tool and the runtime share one source of truth, and fails loudly if the constant can't be found.
- **What gets inlined per page:** a back-to-home link (no site nav exists without JS — the loader injects the sidebar/top bar too), kicker, `<h1>` title, description, the cover image, a `<dl>` metadata grid (Tools / Languages / Period / Role / Tags resolved from the taxonomy), **every** `projectDetails.blocks` entry rendered to its semantic equivalent (`h2/h3/p` as-is; `image` → `<figure><img>` with alt; `image-compare` → a `<figure>` of both before/after images + caption; `video` → a text link to the source URL), and a closing note explaining the interactive version needs JS. This is substantial, real, crawlable body text (e.g. division-2 inlines 40 blocks / 23 figures).
- **Added a `.project-noscript` style block to `style.css`** (≈155 lines, design-token-based: `--text/--muted/--accent/--accent-2/--panel/--panel-2/--line`, responsive at ≤640px) so the text fallback reads like the rest of the site for no-JS visitors. It only ships in `style.css` (always loaded); it is inert for JS visitors because `<noscript>` content is never rendered when scripting is on.

### Design decision — `<noscript>`, not primary visible `<body>`
The review's option (a) literal text says "inline into the page `<body>`." A primary-body pre-render (content inside `[data-project-instance-root]`, which the loader clears on success) would carry marginally more SEO weight, but it has two failure modes I **cannot verify without a browser** (there is no build/test system per `AGENTS.md`, no jsdom/headless browser): (1) a flash of unstyled pre-rendered content for JS visitors during the async fetch window (content appears, is wiped, then re-rendered — worse than the current blank→render), and (2) any edge case where the content isn't cleared risks duplicating it in the rendered DOM. The `<noscript>` approach has **none** of these risks: the HTML5 parser treats `<noscript>` content as RAWTEXT when scripting is enabled (it is never added to the DOM as elements, so it cannot duplicate, flash, or interfere with the loader's `querySelector`/`innerHTML`), and renders it as normal flow content for no-JS visitors and non-JS crawlers. Its only failure mode is "does nothing extra" — the principled call when you can't smoke-test. Google indexes `<noscript>` content; combined with the existing per-page meta + JSON-LD, the pages now have comprehensive static SEO coverage. A primary-body SSR/hydration refactor remains the longer-term enhancement (noted in P1 #1).

### Correctness — bug found and fixed during the build
